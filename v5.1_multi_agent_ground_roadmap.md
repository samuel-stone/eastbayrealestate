# v5.1 Roadmap: Grounding First, Then Parallel Agents

**Project:** East Bay Real Estate Autonomous Agent System  
**Current Version:** v5.0.2  
**Target Evolution:** v5.1 Multi-Agent Architecture  
**Date:** 2026-07-22

---

# Executive Summary

The current autonomous agent pipeline has reached an important transition point.

The system successfully performs:

- automated post-commit analysis
- AI-generated architecture proposals
- code review generation
- PostgreSQL agent memory persistence
- Ollama-powered local inference
- automated Git hook execution

However, before expanding into a true multi-agent architecture, the system needs stronger grounding and verification.

Recent debugging revealed that the primary risk is not lack of intelligence — it is lack of verification.

Examples discovered:

- Hardcoded fake diff content appearing in the proposals UI
- Fabricated performance metrics (`18.4ms avg query speedup`)
- `unhashable type: 'RealDictRow'` failures burning thousands of attempts
- False-positive security findings around `DATABASE_URL`
- AI reviewers making claims without requiring direct evidence

Adding more agents before solving these issues would create more opinions, not more reliability.

The recommended path:

> Build a trustworthy single-agent foundation first, then scale into multi-agent architecture.

---

# Phase 1 — Grounding Layer

## Goal

Make every AI output traceable, verifiable, and structurally reliable.

No agent finding should enter memory unless it contains evidence.

---

# 1.1 Fix UI and Logging Grounding Issues

## Problems

Current system contains places where generated output cannot be distinguished from real system data.

### Proposal UI

Replace hardcoded proposal examples:

```python
st.code("""
--- Target:
core_engine.py
...
""")
```

with actual database-backed values:

```python
obs.get("diff")
```

Fallback:

```
No diff included
```

---

## Remove Fake Metrics

Current:

```python
st.metric(
    "Pipeline Optimization Impact",
    "18.4ms avg query speedup"
)
```

Problem:

The metric has no verified source.

Replace with:

- calculated values from execution logs

or:

- remove entirely

---

## JSON Safety

Current issue:

AI output can contain:

- quotes
- newlines
- markdown
- unexpected characters

Avoid:

```python
f'{"type":"proposal","summary":"{summary}"}'
```

Use:

```python
json.dumps({
    "type": "proposal",
    "summary": summary
})
```

This prevents malformed `agent_memory` rows.

# 1.2 Evidence Requirement

## Current Problem

The AI reviewer currently reasons from:

```
git diff
+
LLM judgment
```

This allowed false positives.

Example:

DATABASE_URL security finding.

The reviewer assumed:

```
environment variable
+
database connection
=
security vulnerability
```

without inspecting actual implementation.

---

# New Finding Contract

Every AI finding must contain:

```json
{
    "severity": "medium",
    "confidence": 80,
    "file": "path/file.py",
    "evidence": {
        "line_start": 120,
        "line_end": 140,
        "snippet": "actual code"
    },
    "problem": "...",
    "recommendation": "..."
}
```

---

# Evidence Gate

Before writing to:

```
agent_memory
```

validate:

```
IF evidence exists:
    store finding
ELSE:
    reject finding
```

Rule:

> No evidence, no memory.

# Implementation Order

| Phase | Objective | Exit Criteria |
|---|---|---|
| Phase 1 | Grounding | Every finding has evidence |
| Phase 2 | Orchestration | Single agent works through framework |
| Phase 3 | Specialists | Multiple evidence-backed agents |
| Phase 4 | Arbitration | Agents disagree and council resolves |

---

# Final Recommendation

The destination is still a multi-agent autonomous engineering system.

The order changes:

Not:

```
More agents → Better intelligence
```

Instead:

```
Grounded evidence
        ↓
Reliable single agent
        ↓
Orchestration
        ↓
Specialists
        ↓
Engineering council
```

The system should earn the right to become multi-agent by first proving that one agent can tell the truth.
